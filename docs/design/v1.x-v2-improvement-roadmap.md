## The one fact that governs everything below

Tagging `v1.0.0` freezes the exported surface of `package mcp` (import path
`github.com/tmc/mcp`, ~411 exported symbols across 36 files) as a compatibility
contract. After the tag, **any change that removes or alters an exported
symbol's signature requires a `/v2` module path** under Go semantic-import
versioning (SIV).

The "this breaks ~50 modules" framing that recurred in review is **overstated**.
Ground truth from the working tree:

- 83 `go.mod` files exist (82 nested + root); 69 reference `github.com/tmc/mcp`.
- **56 of those reach the root via a `replace github.com/tmc/mcp => ../..`
  directive** pointing at the working tree; **0 require a published version
  without a `replace`.** `exp/go.mod` uses `replace ... => ..`; the
  conformance-server has no `go.mod` and builds under the root module itself.
- Every in-repo consumer therefore tracks the **working tree**, not a tag. A
  `v1.0.0` tag does not flow into any of them; they break (or not) the instant
  source changes, tag or no tag — and they are editable in the same commit.
- Grep finds **zero** qualified external uses of `mcp.MCPMethod`,
  `mcp.MCP{Handler,Request,Response,HandlerFunc}`, `mcp.NewEnhancedServer`,
  `mcp.EnhancedServer`, or the server-side `mcp.NotificationHandler`. The only
  live external accessor/option uses are `req.GetMethod()`
  (`exp/cmd-experimental/mcp-health`), `req.GetProgressToken()` and
  `mcp.StreamableHTTPOptions{}` (conformance-server) — all `replace`-bridged.

**Consequence:** the cost of the breaking renames below is almost entirely the
cost of protecting *hypothetical future out-of-tree pinners* — which is exactly
the contract the tag creates. The interface-method renames (`de-get-accessors`)
are the one class that genuinely **cannot be aliased** and must move atomically
at a v2 boundary.

## Pre-tag: cheap, principled, permanent-if-deferred

These two block nothing structural but should land before the tag because their
cost becomes permanent the moment `v1.0.0` exists. Neither re-opens the v1 gate;
both are mechanical and verified zero-external-blast-radius.

### P1 — Delete orphaned `AppleServerOptions`

- **Current state:** `platform_darwin.go:159` declares `type AppleServerOptions
  struct{...}`. A repo-wide grep returns **only its own declaration** — nothing
  constructs or consumes it. The actual Apple knob is the functional option
  `WithAppleOptimizations() ServerOption` (`platform_darwin.go:166`), which does
  not reference the struct.
- **Fix:** delete the type. One-line change, zero blast radius today.
- **Why now:** exporting dead code into a frozen surface is pure liability with
  zero upside. After the tag, removing it is technically breaking (for nobody).

### P2 — `MCPMethod` → `Method` (with deprecation alias)

- **Current state:** `types.go:86` `type MCPMethod string`; the ~30 `Method*`
  constants (`MethodInitialize`, …, `types.go:89-121`) are already typed as
  `MCPMethod` and already read `mcp.MethodInitialize` at the call site. The only
  **exported** signature taking the type is `Dispatcher.NotifyListChanged`
  (`dispatcher.go:59`); the rest are unexported (`server.go:1171`/`1184`) or
  string conversions. `Method` is verified **free** as a top-level type in the
  root package, and there are **zero** qualified `mcp.MCPMethod` uses anywhere.
- **Fix:** rename the type to `Method`; in the same commit add `type MCPMethod =
  Method` marked `// Deprecated:` so any out-of-tree pinner compiles unchanged.
  Drop the alias at v2.
- **Why now:** `mcp.MCPMethod` is textbook stutter (`bytes.Buffer`, not
  `bytes.BytesBuffer`). `net/http` models this correctly with `MethodGet`-style
  constants beside the request surface. Fixing it now avoids ever cutting a
  `/v2` *solely* to correct a name.
- **Caveat to keep crisp:** `Method` is also a common field name
  (`JSONRPCRequest.Method`, `ParameterError.Method`). The type's doc comment must
  distinguish the wire-method-name type from those string fields.

## v1.x: additive — improve the API without breaking it

These land after the tag with **no `/v2`**. They also double as the staging
shims that make the eventual v2 a non-event for downstreams.

### V1.x-1 — `Server.Use` + chain-aware `Serve` (the one that fixes a real bug)

- **Current state — middleware silently does nothing:** `EnhancedServer`
  (`middleware_integration.go:102`) embeds `*Server` and builds a
  middleware-bearing handler at construction
  (`middleware_integration.go:132`, `enhanced.handler =
  enhanced.createUnifiedHandler()`). But `EnhancedServer` has **no `Serve`
  override**, so `enhancedServer.Serve(...)` dispatches to the embedded
  `Server.Serve` (`server.go:1192`), which builds its handler from
  `s.handleRequest` (`server.go:1208`) and **never consults `enhanced.handler`
  or the middleware manager.** Middleware configured via
  `SetMiddlewareConfig`/`UseMiddleware` never runs on real traffic. The lone
  integration test (`middleware_test.go:727`) only asserts `globalChain != nil`
  and comments "in practice, you'd test the full request flow" — it never drives
  a request through the chain.
- **Fix (additive):** give `Server` an optional middleware chain field and an
  exported `func (s *Server) Use(m Middleware)` (net/http-style); in `Serve`
  (`server.go:1208`) wrap the handler so the chain runs per request. Deprecate
  `EnhancedServer` in a doc comment pointing at `NewServer`+`Use`.
  ```go
  // Before — never runs:
  server := NewEnhancedServer()
  server.UseMiddleware(NewLoggingMiddleware(...))
  server.Serve(ctx, t)

  // After — runs on every request:
  s := NewServer("name", "v1")
  s.Use(NewLoggingMiddleware(...))
  s.Serve(ctx, t)
  ```
- **Why:** "the bigger the interface, the weaker the abstraction." A parallel
  type that type-checks, documents a capability, has passing tests, and does
  nothing on the wire is the worst kind of API. This is the part worth doing;
  the constructor rename (V2-3 below) is the cosmetic symptom.
- **Do not** ship the rename without this wiring, or you have renamed dead code.

### V1.x-2 — Ctx-first `NotificationHandlerFunc` (fixes a dropped-context bug)

- **Current state — context is received then discarded:** `types.go:651`
  `type NotificationHandler func(method string, params json.RawMessage) error`
  is the only member of the handler family that omits both the `…Func` suffix
  and a leading `context.Context` (compare `ToolHandlerFunc`,
  `GetPromptHandlerFunc`, … at `types.go:653-666`, all `ctx`-first). Worse,
  `Dispatcher.Dispatch` already **has** a ctx (`dispatcher.go:34`) and **drops
  it** at `dispatcher.go:48` (`handler(method, params)`), so notification
  handlers cannot observe cancellation or deadlines. The project's own next-gen
  line already converged on the right shape:
  `exp/sdk2/types.go:743` `NotificationHandlerFunc func(context.Context, string,
  json.RawMessage) error`.
- **Fix (additive):** introduce a new ctx-first `NotificationHandlerFunc`; add
  `Dispatcher.HandleFunc(method, NotificationHandlerFunc)`; have `Dispatch` try
  both type-assertions (old without ctx, new with ctx) and forward the ctx it
  already holds. Mark the old type `// Deprecated:`.
  ```go
  type NotificationHandlerFunc func(ctx context.Context, method string, params json.RawMessage) error
  ```
- **The dual assertion is mandatory, not optional:** the dispatcher stores
  handlers as `map[string][]any` (`dispatcher.go:18`) and recovers them via
  runtime type assertion (`dispatcher.go:44`). A bare rename means any handler
  registered with the old shape **silently fails the assertion and is skipped**
  — a silent runtime regression (notification undelivered), not a build error.
  The dual-assertion shim is what keeps that from happening across the
  deprecation window.
- **Note:** the client-side notification callback is a *different* shape
  (`client.go:39`/`:119` `func(notification JSONRPCNotification)`) — a separate
  inconsistency, out of scope here.

### V1.x-3 — SSE server handler (defer to v1.1, not v1.0)

- **Current state — the remote-transport surface is asymmetric:** the root has
  client-side SSE (`SSEClientTransport`, `transport_sse.go:32`) and WebSocket
  (`WebSocketTransport`, `transport_websocket.go:15`), and a server-side
  Streamable HTTP handler (`StreamableHTTPHandler.ServeHTTP`,
  `transport_streamable.go:178`). But the only server-side SSE type,
  `SSEServerTransport` (`transport_sse.go:333`), is **not** an `http.Handler` —
  it wraps a pre-existing `io.ReadWriteCloser`, and its `Dial` returns
  `"SSEServerTransport: RWC not initialized by HTTP handler"`
  (`transport_sse.go:348`), naming a handler that does not exist in the package.
  There is no SSE `ServeHTTP` and no `websocket.Upgrader` in the root (the only
  upgraders live in `exp/` tools). CI interop (`ci.yml:533-540`) covers stdio +
  Streamable HTTP only.
- **Fix (additive):** add `NewSSEHandler(getServer func(*http.Request) *Server,
  opts *SSEHandlerOptions) *SSEHandler` with a real `ServeHTTP`, mirroring
  `NewStreamableHTTPHandler` so the two compose under one mux. It feeds an
  `io.ReadWriteCloser` into the existing `SSEServerTransport`, removing the dead
  error path at `transport_sse.go:348`. Add an `sse_smoke.mjs` interop fixture
  and matching test; correct the now-false `ci.yml:536-537` comment claiming
  stdio+streamable are "every transport the root package can serve."
- **Why v1.1, not v1.0:** purely additive (no `/v2` ever needed), so SIV imposes
  no deadline. A correct SSE handler needs session handling, flush/keepalive,
  backpressure, and graceful shutdown to match the Streamable handler's bar
  (~860 lines for a reason). Don't rush new untested-in-the-wild server code into
  the frozen v1.0.0 surface. **Alternative honest pre-tag move:** instead of
  completing it, document `SSEServerTransport` as bring-your-own-handler (or
  deprecate it) so v1.0.0 doesn't ship a sharp unfinished edge.

### V1.x-4 — Move `ResponseError` to `errors.go`

- **Current state:** `ResponseError` (a JSON-RPC protocol error type) lives in
  `middleware.go:695`, an accidental home next to the middleware interfaces.
- **Fix:** move the declaration to `errors.go` (or `types.go`) beside
  `JSONRPCRequest`/`Response`/`Notification`. Internal file move, no API change,
  zero blast radius. Do it whenever convenient.

## v2: batch every breaking change into ONE designed release

Each item below is individually a `/v2` event under SIV. **Do not dribble them
as `/v2`, `/v3`, `/v4`.** A user migrating across a module-path bump pays a
fixed, irreducible cost: edit every import, re-resolve, re-test — *the same cost
whether v2 changes one symbol or twenty*. Dribbling multiplies a one-time
migration tax by the number of releases for zero benefit. Accumulate these in a
`v2-TODO.md`, design the v2 surface as one coherent revision, cut `/v2` once.

The deepest v2 design question — settle it before cutting — is the **two
parallel handler abstractions**: `typed.go`'s generic `Handler`/`HandlerFunc`
(`typed.go:290`/`:295`) coexisting with the middleware `MCP*` handler family
(`middleware.go:47-68`). v2 should consolidate these, not just rename around them.

### V2-1 — De-`Get` the pure accessors (cannot be aliased; v2-only)

- **Current state:** Effective Go says accessors drop the `Get` prefix
  (`owner.Name()`, not `owner.GetName()`; stdlib: `bytes.Buffer.Len`,
  `time.Time.Hour`, `reflect.Value.Kind`). Violated on pure field reads:
  - `MCPRequest`: `GetMethod`/`GetID`/`GetParams`/`GetContext` (`middleware.go:52-58`)
  - `MCPResponse`: `GetResult`/`GetError` (`middleware.go:61-64`)
  - `Progress`: `GetValue`/`GetMessage`/`GetTotal` (`progress.go:70-88`)
  - `CallToolRequest.GetProgressToken` (`types.go:344`)
- **Keep `Get` where it does real work (leave alone):** `OAuthProvider.GetClient`
  / `GetAuthorizationCode` (`auth.go:133`/`:138`) are ctx+error store lookups
  (analogous to `http.Get`), not field reads. `Client.GetPrompt`
  (`client.go:323`) tracks the wire verb `prompts/get` — renaming would collide
  with the `Prompt` type and obscure the mapping.
- **Fix:** `GetMethod`→`Method`, `GetID`→`ID`, `GetParams`→`Params`,
  `GetContext`→`Context`, `GetResult`→`Result`, `GetError`→`Error`,
  `GetValue`→`Value`, etc. Verified no method/field name collisions (the concrete
  types — `UnifiedRequest`, `Progress` — use unexported fields).
- **Why v2-only:** these are **interface methods**, which Go cannot alias. They
  must move atomically at the v2 boundary. The struct accessors (`Progress`,
  `CallToolRequest`) *could* add non-`Get` methods in v1.x and deprecate the
  `Get` ones, but the interface methods force the batch.
- **Secondary smell to weigh in the v2 redesign:** `MCPRequest` stores a
  `context.Context` and exposes it via `GetContext()`/`WithContext()` — counter
  to "pass ctx as the first arg." And `MCPResponse.Result()` returns
  `interface{}` regardless of prefix. A v2 may want to reshape these adapters
  (or unexport them) rather than merely de-`Get` them.

### V2-2 — `StreamableHTTPOptions` / `StreamableClientTransportOptions` → `…Config`

- **Current state:** the package has **18 exported `*Config`** structs and **3
  `*Options`** structs. The two streamable `*Options`
  (`transport_streamable.go:49`, `transport_streamable_client.go:18`) are
  role-identical to the `*Config` types — passive bags passed to constructors,
  their own doc comments say "configures …". Meanwhile `Option` is *already*
  claimed by the genuine functional-option mechanism (`ServerOption`,
  `ClientOption` — `options.go:9`, `client.go:33`). So a reader sees
  `ServerOption` (a func) and `StreamableHTTPOptions` (a passive struct): same
  root word, two unrelated mechanisms.
- **Rule to adopt** (matches 18/21 existing types and stdlib `http.Server`,
  `tls.Config`, `http.Transport`): passive struct passed to a constructor →
  `*Config`; functional option `func(*T)` → `*Option`.
- **Fix:** `StreamableHTTPOptions`→`StreamableHTTPConfig`,
  `StreamableClientTransportOptions`→`StreamableClientConfig`, update the two
  constructors. **Can be v1.x-staged** via `type StreamableHTTPConfig =
  StreamableHTTPOptions` aliases + deprecation, dropped at v2.
- **Check first:** if the `Options` names deliberately echo the upstream
  `modelcontextprotocol/go-sdk` surface for migration parity, that's a reason to
  keep them. Verify before renaming.

### V2-3 — Remove the `EnhancedServer` facade type

- **Current state:** `NewEnhancedServer(opts...)` (`middleware_integration.go:112`)
  and `NewEnhancedServerWithName(name, version, opts...)`
  (`:117`) are a redundant pair; `NewEnhancedServerWithName` has **no callers**.
  And per V1.x-1, the type's whole reason to exist (middleware) doesn't run.
- **Fix:** once `Server.Use` lands (V1.x-1), delete `EnhancedServer`,
  `NewEnhancedServer`, `NewEnhancedServerWithName`, and demote/remove the
  `TransportMiddlewareManager` facade. If any facade survives, collapse to a
  single `NewEnhancedServer(name, version, ...opts)` matching the `NewServer`
  convention (there is no `NewServerWithName`; don't introduce a `*WithName`
  sibling anywhere). Blast radius: these symbols have **no importers outside the
  root package**.

### V2-4 — Remove the deprecated `NotificationHandler` and `MCPMethod` aliases

Once the v1.x deprecation windows (V1.x-2, P2) have shipped: drop the deprecated
`NotificationHandler` type and collapse `Dispatch` to a single assertion; drop
`type MCPMethod = Method`.

## Declined — explicitly NOT debt

Recording these so a future reviewer doesn't re-litigate settled questions.

### D1 — `NewServer` error-return symmetry → **working as intended**

`NewServer` (`server.go:154`) is infallible: maps + logger + options, no I/O,
and `ServerOption = func(*Server)` (`options.go:9`) cannot even *express*
failure. `NewClient` returns an error *because it dials at construction*
(`client.go:74`). That asymmetry is exactly the `bytes.NewBuffer`-vs-`os.Open`
distinction, applied correctly across all ~30 root constructors. Adding an
always-nil error would force a dead `if err != nil` on every caller. **Do not
"fix" this.** If a future `ServerOption` ever needs to fail, stash the error and
surface it from `Serve`, or add a sibling constructor — don't change
`NewServer`'s signature.

### D2 — Split the flat root into subpackages → **keep flat**

`net/http` is the yardstick and it is flat: 34 non-test files, ~30k LOC, holding
client + server + `Transport` + TLS in one package. `mcp` at 36 files / ~15.5k
LOC is *smaller* and is exactly the single-protocol library Go keeps flat on
purpose. The coupling is real and bidirectional (`EnhancedServer` embeds
`Server`; `Server.Serve` takes `Transport`; ratelimit/auth implement the
middleware interfaces), so any split just relocates the import cycle into a
shared base package — the classic sign the boundary is artificial. The only
genuine smell is the misfiled `ResponseError` (a file-organization issue → V1.x-4,
zero cost), not a topology one. The 411-symbol godoc scroll is a real ergonomic
cost, but not one that justifies a `/v2` and cross-repo import churn.

### D3 — WebSocket server upgrader → **decline (not a blessed transport)**

WebSocket has never been an official MCP transport, and the spec's 2025-03-26
revision *deprecated* standalone HTTP+SSE in favor of Streamable HTTP. Building a
WS upgrader implements a transport the protocol does not bless. Streamable HTTP
is the supported remote transport; `WebSocketTransport` stays client-or-BYO-handler.

### D4 — Middleware `MCP*` interface renames as a standalone change → **decline**

Renaming `MCPHandler`→`MiddlewareHandler` (etc.) papers over but does not resolve
the deeper smell: two parallel handler abstractions (`typed.go`'s generic
`Handler`/`HandlerFunc` vs the middleware `MCP*` family). A cosmetic prefix-strip
that leaves the duplication intact is bikeshedding. Fold it into the v2
handler-consolidation (see the V2 preamble), or do nothing.
